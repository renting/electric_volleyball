## Context

`electric_volleyball.html` 是單一 HTML 檔案的 Phaser 3 遊戲，物理為自製逐幀計算（非 Arcade Physics）。現況：

- 玩家移動速度每幀 5.4 px（`_controlHuman` 的 `SPD`），唯一主動擊球是 `_tryPowerHit`，判定範圍為 `Math.hypot(dx, dy) < p.r + ball.r + 24`（以 `p.y - p.r*0.4` 為玩家判定中心）。
- 按扣殺鍵但球不在範圍內時，`_tryPowerHit` 的 else 分支設定 `p.hitCooldown = 10`（揮空冷卻）。
- 1P 模式的扣殺鍵（Space/Z）目前不經 `_controlHuman` 的 `hit` 參數，而是在 `update()` 內獨立判斷；2P 模式則由 `_controlHuman(p, ..., hit)` 處理（P1 `G`、P2 `L`）。
- 玩家與球的碰撞在 `_updatePhysics()` 中以單一圓（中心 `p.y - p.r*0.4`、半徑 `p.r + b.r`）計算。
- 角色繪製集中於 `drawMonGraphics(g, player)`，已有依 `onGround` 的 squash 形變機制。

技術約束：

- 維持單一 HTML 檔案、無新增相依。
- 維持自製逐幀物理；撲球必須以同樣的「狀態欄位 + 每幀步進」模式整合（同 `hitCooldown`、`smashFlash`）。
- 球在扣殺範圍內時的既有扣殺行為不得改變。

## Goals / Non-Goals

**Goals:**

- 扣殺鍵分流：球在範圍內 → 扣殺（不變）；球在範圍外 + 按住單一方向鍵 + 在地面 → 撲球；其餘 → 揮空冷卻（不變）。
- 撲球期間身體扁平延伸、貼地高速位移，碰到球做出「墊球」式高彈回擊。
- 撲球結束後有趴地起身硬直，期間輸入鎖定。
- 1P 與 2P 的人類玩家共用同一套撲球邏輯。
- 更新 `#hint` 與 MenuScene 操作提示。

**Non-Goals:**

- AI（單人模式 P2）不實作撲球。
- 不支援空中撲球（僅 `onGround` 可觸發）。
- 不新增按鍵、不改變既有鍵盤配置。
- 不調整扣殺、移動、跳躍、AI 難度等既有手感參數。
- 不引入 Phaser Tween 或 Arcade Physics。

## Decisions

### 決策一：扣殺鍵分流放在控制層，並統一 1P/2P 路徑

**選擇**：抽出共用判定函式 `_inHitRange(p)`（回傳 `Math.hypot(dx, dy) < p.r + ball.r + 24`，判定中心與 `_tryPowerHit` 完全一致），分流邏輯寫在 `_controlHuman` 內：

```js
if (hit && hit.isDown && p.hitCooldown <= 0) {
  if (this._inHitRange(p))                    this._tryPowerHit(p);   // 行為不變
  else if (this.state === 'play' && p.onGround && dirHeld !== 0)
                                              this._startDive(p, dirHeld);
  else                                        p.hitCooldown = 10;     // 揮空，行為不變
}
```

1P 模式改為把 `Space`/`Z` 合併成一個虛擬 hit 狀態（`{isDown: space.isDown || z.isDown}`）傳入 `_controlHuman`，移除 `update()` 內的獨立判斷，讓 1P/2P 走同一條路徑，分流邏輯只存在一份。

`dirHeld` 取值：只按左 → `-1`、只按右 → `+1`、都沒按或同時按住左右 → `0`（不觸發撲球，進揮空冷卻）。

撲球僅限 `play` 狀態：控制路徑在 `serve` 倒數期間也會執行，而此時球凍結在 `y = 110` 的遠處、必定不在扣殺範圍內，若不加狀態條件，倒數中按扣殺＋方向鍵必定觸發撲球，且倒數尾端起跳的撲球會把約 30 幀的輸入鎖定帶進回合開始。故 `serve` 期間維持既有揮空行為。

**放棄方案**：在 `_tryPowerHit` 內加參數分流 — 會讓單一函式同時負責扣殺與撲球兩種動作，且 `_tryPowerHit` 的 else 分支語意（揮空）會被覆寫，職責混亂。

### 決策二：撲球以狀態欄位 + 每幀步進實作

**選擇**：player 物件新增三個欄位，與既有 `hitCooldown`/`smashFlash` 同一模式：

| 欄位 | 意義 |
|------|------|
| `diveT` | 撲球位移剩餘幀數（`0` = 非撲球中） |
| `diveDir` | 撲球方向（`-1` \| `1`） |
| `diveRecover` | 起身硬直剩餘幀數（`0` = 非硬直中） |

步進在 `_updatePhysics()` 的玩家迴圈中執行；`_controlHuman` 開頭檢查 `diveT > 0 || diveRecover > 0` 時直接 return（輸入鎖定）。`_resetRally()` 清除兩名玩家的 `diveT`/`diveRecover`，避免回合切換時殘留撲球狀態。

**放棄方案**：Phaser Tween 做位移 — 與自製物理的逐幀邊界 clamp、暫停旗標（`paused` 時 `update()` 早退）不相容，會在暫停時繼續位移。

### 決策三：撲球參數

**選擇**（集中為常數，便於 playtest 調整）：

| 常數 | 值 | 說明 |
|------|----|------|
| `DIVE_TIME` | 14 幀 | 位移階段長度 |
| `DIVE_SPD` | 9 px/幀 | 位移速度（總位移 126 px ≈ 普通移動 23 幀的量） |
| `DIVE_RECOVER` | 16 幀 | 起身硬直長度 |

**理由**：撲球是「賭注式」救球——14 幀內獲得約 1.67 倍移動速度與更遠的判定範圍，代價是後續 16 幀完全無法行動（總承諾約 0.5 秒 @60fps）。撲空的懲罰要明顯大於揮空（10 幀），否則撲球會取代普通移動。

位移期間 `p.x` 仍套用既有邊界 clamp（網側 `NET_X ± (NET_W/2 + p.r)`、牆側 `p.r`/`W - p.r`）；撞到邊界時位移停止但 `diveT` 照常遞減。

### 決策四：撲球判定區以「前移 + 貼地」單圓近似

**選擇**：`diveT > 0 || diveRecover > 0` 期間，玩家與球的碰撞圓心由 `(p.x, p.y - p.r*0.4)` 改為 `(p.x + diveDir*p.r*0.7, p.y - p.r*0.5)`，半徑維持 `p.r + b.r`。

**理由**：圓心前移 `0.7r` 後，朝撲球方向的有效觸及達 `x + 1.7r + b.r`，與「身體水平延伸 1.6 倍」的視覺大致吻合；貼地（`p.y = GROUND` 時圓心約在地面上方 `0.5r`）符合趴地姿態。單圓沿用既有碰撞數學，不需新增橢圓/膠囊碰撞。

**自己半場限制**：撲球判定區僅對「球心在自己半場」的球生效（side 0：`ball.x < NET_X`；side 1：`ball.x > NET_X`），球心在對側時該幀改用既有站立判定圓。否則朝網撲球被 clamp 擋下時，前移 `0.7r` 的判定圓會穿過網子搆到貼網的對側球——以左方為例：網側 clamp 在 `x = 411`，撲球圓心 `434.8`，觸及達 `x ≈ 486`，而被網側推開的對側球心在 `472`（站立判定圓搆不到，距離約 61 > 51），等於新增了隔網撈球的能力。

**放棄方案**：橢圓或雙圓精確碰撞 — 精度提升對 17px 的球感知有限，卻要改寫碰撞推出與法線計算。

### 決策五：墊球（dig）回擊固定高彈，碰球即轉入起身

**選擇**：`diveT > 0`（位移階段）碰到球時，先沿法線推出重疊（與既有碰撞相同），但速度改為固定式墊球回擊：

```js
b.vy   = -11;                          // 強制高彈起
b.vx   = p.diveDir * 3 + b.vx * 0.25;  // 水平大幅減弱、略偏撲球方向
b.spin = p.diveDir * 0.25;
p.smashFlash = 8;                       // 較淡較短的光環（沿用既有光環公式，alpha 較低、壽命較短）
p.diveT = 0; p.diveRecover = DIVE_RECOVER;  // 碰球即起身
p.hitCooldown = DIVE_RECOVER;               // 單次觸球：起身期間不再與球碰撞
if (b.y > GROUND - b.r - 1) b.y = GROUND - b.r - 1;  // 墊球優先於同幀落地判定
```

兩條補強規則的由來：

- **單次觸球（`hitCooldown` 閘門）**：撲球狀態（`diveT > 0 || diveRecover > 0`）的碰撞判定僅在 `p.hitCooldown <= 0` 時進行；墊球將其設為 `DIVE_RECOVER`，恰好覆蓋整段起身（每幀遞減，起身結束時歸零）。若無此閘門，推出重疊後球恰好貼在判定圓邊界，自後緣墊到的球（例如從背後追上、`vx > DIVE_SPD` 的快球）下一幀會再觸發起身期的一般碰撞，把向上墊起的球反向踢飛並重複觸發粒子與音效。撲球觸發已要求 `hitCooldown <= 0`，撲球期間輸入鎖定也不可能再揮空，故此閘門不會誤擋正常撲球碰撞。
- **墊球優先於落地判定**：撲球判定圓心高度（`GROUND - 0.5r = 437`）恰等於球的落地門檻（`GROUND - ball.r = 437`），且玩家碰撞迴圈在落地判定之前執行；不修正 `b.y` 時，「落地幀」墊到的球推出後仍可能 `b.y >= GROUND - b.r`，同幀觸發得分——玩家看到墊球特效卻被判失分，球還會卡在沙面下直到下一回合。修正後成功墊球一律優先於落地。

並觸發回饋：`_burst(b.x, b.y, 0xfff3b0, 10)`、`beep(480, .08, 'triangle', .12)`。

`diveRecover > 0`（起身階段）碰到球時：位移自然結束（未墊球）的起身沿用既有的一般碰撞速度轉移（僅圓心位置不同，含自己半場限制），不再觸發墊球；墊球後的起身因 `hitCooldown > 0` 不與球碰撞。

**理由**：一般碰撞的速度轉移方向取決於法線，從低處撲到球時球常會被打向地面，失去「救起來」的意義。固定 `vy = -11` 保證球向上飛，水平減弱讓墊起的球大多留在己方場地，符合排球「先救起再組織」的節奏。碰球即轉入起身，配合 `hitCooldown` 閘門確保同一次撲球只觸球一次。

**放棄方案**：沿用一般碰撞速度轉移 — 無法保證救球向上；墊球後球可能直接出界或下網，玩家會覺得撲球無用。

### 決策六：撲球姿態繪製整合進 drawMonGraphics

**選擇**：`drawMonGraphics` 依 dive 狀態計算形變參數，線性過渡：

- 位移階段（`diveT > 0`）：身體橢圓水平直徑 ×1.6、垂直直徑 ×0.5，身體中心移至 `(x + diveDir*r*0.45, GROUND - r*0.55)`，`face = diveDir`。
- 起身階段（`diveRecover > 0`）：令 `t = diveRecover / DIVE_RECOVER`（1 → 0），形變參數從撲球姿態線性插值回站姿（水平 `1 + 0.6t`、垂直 `1 - 0.5t`、中心水平偏移 `diveDir*r*0.45*t`、中心高度同步插值；四項皆插值，避免任一參數在 `t = 0` 時跳變）。
- 尾巴、耳朵、眼睛、臉頰、嘴巴等元素以新的身體中心為基準重新定位，元素不得缺漏。
- 地面投影橢圓寬度隨水平延伸同步加寬。

**放棄方案**：用整體旋轉 90° 的站姿表現趴地 — Phaser Graphics 沒有便捷的整體旋轉（需逐點變換），且旋轉站姿視覺僵硬，不符「扁平延伸」的需求。

## Risks / Trade-offs

- **平衡風險**：`DIVE_SPD`/`DIVE_RECOVER` 設定不當會讓撲球取代移動（太強）或無人使用（太弱）。→ 參數集中為常數，tasks 安排實際遊玩驗證（含故意撲空確認懲罰感）。
- **連續撲球**：扣殺鍵與方向鍵按住不放時，硬直結束的下一幀若球仍在範圍外會立即再次撲球（`isDown` 語意）。→ 暫定視為特性（連滾翻撲球），與既有「按住扣殺鍵進範圍即自動扣殺」的 isDown 語意一致；若 playtest 覺得過強，改為 `Phaser.Input.Keyboard.JustDown` 觸發即可，調整點只有一處。
- **起身階段視覺與判定的不一致**：起身的視覺是漸變回站姿，但判定圓固定維持「前移 + 貼地」直到 `diveRecover` 歸零。→ 接受此簡化；硬直僅 16 幀，且判定區偏小（貼地）對玩家是保守誤差。
- **撲球撞網**：朝網撲球會被既有邊界 clamp 擋下，位移浪費但判定圓仍前移 `0.7r`，可救己方網前球；決策四的「自己半場限制」防止判定圓隔網搆到對側貼網球。→ 屬預期行為。

## Open Questions
